# Anthropic：用平行 Claude 打造 C 編譯器

> 原文：<https://www.anthropic.com/engineering/building-c-compiler>  
> 發布日期：2026-02-05  
> 作者：Nicholas Carlini  
> 相關程式碼：<https://github.com/anthropics/claudes-c-compiler>

> 本文為依原始文章內容整理、重述並翻譯的繁中技術筆記，採台灣常用技術術語，不是逐段全文直譯。

## 核心結論

Anthropic 用一組可平行運作的 Claude agent team，嘗試從零開始寫出一套以 Rust 實作的 C 編譯器。這個實驗的重點不只是「模型會寫編譯器」，而是展示長時間自主執行、多 agent 平行協作、以及測試框架如何決定最終成果。

作者的觀察很直接：真正的瓶頸不是讓模型「一直跑」，而是要先把環境搭好，讓模型知道自己下一步該做什麼、怎麼驗證、出錯時去哪裡找訊號。

## 這個實驗做了什麼

- 使用 16 個 Claude agent 平行協作。
- 累積接近 2,000 次 Claude Code session。
- API 成本接近 20,000 美元。
- 產出約 10 萬行 Rust 程式碼。
- 最終做出一套可編譯 Linux 6.9 的 C 編譯器。
- 目標架構包含 x86、ARM 與 RISC-V。

根據文章與公開儲存庫，這套編譯器可以處理不少真實專案，並在多數編譯器測試集上達到約 99% 通過率。不過它仍不是成熟商用等級的 GCC 替代品。

## 長時間自主執行的關鍵：無限迴圈執行框架

作者先做了一個很簡單的執行框架（harness）：

1. 啟動 Claude Code。
2. 丟入固定的 agent prompt。
3. Claude 完成一輪後，立刻開下一輪。
4. 這個流程持續重複，直到外部停止。

這樣做的目的，是避免單一 session 解完一半就停下來等人類回覆。作者等於把 Claude 放進一個「做完一件事就繼續下一件事」的自動循環裡。

作者也特別提醒，這種做法應該跑在容器裡，而不是直接跑在自己的主機上。

## 多個 Claude 怎麼平行協作

這套平行化方法本身不算複雜，核心是 Git 儲存庫加上任務鎖定：

1. 建立一個共享的 bare Git repository。
2. 每個 agent 在自己的 Docker 容器裡工作。
3. 每個容器把共享版本庫掛載進去，再 clone 一份本地工作目錄。
4. agent 先透過 `current_tasks/` 裡的鎖定檔宣告自己要處理哪個任務。
5. 做完後先 pull、合併其他 agent 的變更、再 push 回上游，最後釋放鎖定。
6. 下一輪再由新的 Claude session 接手後續工作。

這裡沒有中央協調器，也沒有一個上層總管 agent 幫大家分派工作。作者是讓每個 Claude 自己判斷「現在最明顯、最值得做的下一件事」。

## 為什麼測試框架比模型本身更重要

作者反覆強調：如果驗證機制不夠準，Claude 就會把錯的事情做得很認真。

因此大部分時間其實花在這些地方：

- 尋找高品質的編譯器測試集。
- 幫開源專案補建置腳本與驗證器。
- 觀察 Claude 常犯的錯，再追加新測試去堵這些失敗模式。
- 在專案後期補上更嚴格的 CI，降低新功能破壞既有功能的機率。

這裡的重點不是「測試越多越好」，而是「測試輸出的訊號必須夠清楚，讓模型知道下一步要修哪裡」。

## 從模型視角設計環境

作者提到一個很重要的觀念：這些測試不是寫給人看的，而是寫給 Claude 用的。

因此環境設計有幾個原則：

### 1. 不要污染 context window

- 測試輸出不能灌一大堆無用日誌。
- 終端輸出只保留少量高價值訊息。
- 詳細資訊應該寫進 log 檔，讓 agent 需要時再去查。
- 錯誤格式要容易被自動處理，例如固定出現 `ERROR` 與錯誤原因。

### 2. 要處理模型的「時間盲」

Claude 不會真的理解自己已經花了多久時間。如果不限制，它可能一直跑完整測試，卻沒真的推進問題。

作者的做法是：

- 預設提供快速模式。
- 只跑 1% 或 10% 的隨機樣本。
- 對單一 agent 來說抽樣結果是固定的，方便定位回歸。
- 不同虛擬機器的抽樣又不一樣，讓整體覆蓋面仍然夠大。

這是一種很務實的工程取捨：先讓 agent 快速獲得可行回饋，再逐步擴大驗證範圍。

## 真正讓平行化成立的方法

當測試集中有很多彼此獨立的失敗案例時，多 agent 很容易分工，每個人撿不同的 bug 修就好。

但到了 Linux kernel 編譯階段，事情變成一個巨大的單一任務：

- 所有 agent 都會撞到同一個錯誤。
- 大家都在修同一塊。
- 最後不斷互相覆蓋彼此的改動。

作者後來引入 GCC 當作「正確性基準（oracle）」來拆解問題：

1. 大部分檔案先用 GCC 編譯。
2. 只讓 Claude 的編譯器負責剩下的一小部分。
3. 如果 kernel 可以動，表示錯誤不在 Claude 負責的那批檔案。
4. 如果壞掉，就再逐步縮小問題範圍。

這樣就把原本單一巨大問題，拆成多個可以平行搜尋的子問題。每個 agent 可以鎖定不同檔案或不同故障來源，而不是所有人卡在同一個地方。

## 多角色 agent 的實際用途

作者也嘗試讓不同 agent 做專職工作，而不是所有 agent 都做一樣的事：

- 有的 agent 專門找重複實作並做整併。
- 有的 agent 專門改善編譯器本身效能。
- 有的 agent 專門讓輸出的機器碼更有效率。
- 有的 agent 站在 Rust 開發者角度批評整體設計。
- 有的 agent 專門維護文件。

這個做法反映一個現象：多 agent 的價值不只在平行處理，也在角色分工與專業化。

## 成果評估

文章列出的主要成果如下：

- 兩週左右的連續實驗。
- 約 20 億 input tokens。
- 約 1.4 億 output tokens。
- 總成本略低於 20,000 美元。
- 僅依賴 Rust 標準函式庫。
- 可建出可開機的 Linux 6.9。
- 可編譯 QEMU、FFmpeg、SQLite、PostgreSQL、Redis 等專案。
- 多數編譯器測試集通過率約 99%。
- 還能成功編譯並執行 Doom。

公開儲存庫 README 也補充，這個專案主打「從前端、SSA IR、最佳化器、程式碼產生器、組譯器、連結器到 DWARF 除錯資訊產生，全都自己做」，不依賴外部編譯器專屬套件。

## 已知限制

這套編譯器雖然驚人，但作者沒有把它描述成已完成產品。文章明講了幾個限制：

- 缺少能在 Linux 開機早期使用的 16-bit x86 編碼器，因此這一段仍會呼叫 GCC。
- 自製 assembler 與 linker 還不夠穩。
- 並不是所有真實世界專案都能成功編譯。
- 輸出的程式碼效率仍明顯落後 GCC。
- Rust 程式碼品質雖然可接受，但距離資深 Rust 工程師水準仍有落差。

作者也提到，到了專案後段，新功能與 bug 修正常常會重新破壞舊功能，表示系統已接近模型能力上限。

## 這篇文章真正想講的事

這篇文章表面上是在展示「Claude 幫忙寫出 C 編譯器」，但實際上更像是在回答下面幾個問題：

- 長時間自主 agent 要怎麼避免停住？
- 多個 agent 怎麼共用同一份程式碼庫而不互相踩壞？
- 怎樣的測試輸出才真的對模型有用？
- 哪些問題適合平行化，哪些不適合？
- 模型能力接近極限時，會怎麼失敗？

作者的立場也很清楚：這種自主開發方式很有潛力，但風險也很真實。若人類沒有親自驗證，就因為測試綠燈而直接相信結果，會是很危險的工程習慣。

## 一句話總結

這篇文章最有價值的地方，不是證明 LLM 可以寫大型系統，而是證明只要測試框架、任務切分、回饋訊號與版本協作設計得夠好，多個自主 agent 已經能把「原本看起來不可能」的軟體工程任務，推到接近可落地的程度。
