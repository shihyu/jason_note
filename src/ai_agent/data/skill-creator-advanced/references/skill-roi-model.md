# Skill ROI model

這份文件用來判斷一個 skill 是否值得長期維護。

## 為什麼要看 ROI

skill 不是越多越好。它有：
- 寫作成本
- 測試成本
- 維護成本
- trigger 衝突成本
- token / 時間成本

如果效益太小，應縮 scope、拆 skill，甚至退回一般 prompt。

## 最低評估面向

### 1) 品質提升
- pass rate 是否提高
- 錯誤率是否下降
- 人工糾正次數是否減少

### 2) 成本增加
- tokens 增加多少
- latency 增加多少
- 是否需要更多 scripts / references / reviewer

### 3) 維護成本
- 需求是否常變
- 是否容易和其他 skill 衝突
- 是否需要頻繁更新 reference

## 常見判準

通常值得保留：
- 明顯降低錯誤率
- 顯著減少人工糾正
- 高風險任務可顯著降低事故

通常不值得保留：
- 只提升一點點品質，但成本大增
- 只對極少數情境有用
- 功能與現有 skill 高度重疊
- 每次改版都要大量手動調整

## 建議問題

發版前至少問：
- 若沒有這個 skill，損失是什麼？
- 這個 skill 是否明顯節省使用者或團隊時間？
- 這個 skill 是否在 3-6 個月後仍值得維護？

## 決策輸出

最後應有一個明確結論：
- 保留並持續投資
- 保留但縮 scope
- 拆成兩個 skill
- 退回一般 prompt / workflow
